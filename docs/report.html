I built an end-to-end data engineering system on 4,225,958 synthetic East African mobile money transactions: a chunked Python pipeline loading data into PostgreSQL, and a Text-to-SQL RAG chatbot — Sele the Analyst — that answers plain-English questions about the data with grounded, hallucination-checked responses and a confidence score. The full stack runs with a single docker compose up command.

1. Approach

Dataset

I selected the MoMTSim Synthetic Mobile Money Transaction Dataset (Kaggle, CC0) — 4,225,958 rows covering five M-Pesa-style transaction types across nine simulation days. This exceeds the 50,000-row minimum by 84× and closely mirrors real East African mobile money patterns.

Data Pipeline — Load · Clean · Transform · Store

The pipeline processes the CSV in 100,000-row chunks using pandas.read_csv(chunksize=100_000), keeping memory flat at ~150 MB regardless of file size. Each chunk is cleaned (null drops, type normalisation, deduplication) and enriched with seven derived columns before being written to PostgreSQL.

Bulk insert uses COPY FROM STDIN via psycopg2.copy_expert — achieving ~80,000 rows/second vs ~1,000 rows/second for row-by-row INSERT. Full load completes in approximately 2 minutes. A transaction_summary pre-aggregation table (45 rows: 9 days × 5 types) is computed once after load, reducing analytical query cost from 4.2M row scans to 45-row lookups.

RAG Application — Sele the Analyst

I chose Text-to-SQL RAG over vector similarity retrieval — the correct pattern for structured relational data. Vector search is designed for unstructured text documents; for financial records in a relational database, the correct flow is: question → SQL → execute → ground the answer in real results. The LLM cannot hallucinate a number because it reads from actual rows.

Question → [LangGraph: classify] ├─ data query → [GPT-4o-mini: generate SQL] → [PostgreSQL: execute] │ → [back-translation: hallucination check] → [GPT-4o-mini: answer] ├─ follow-up → [GPT-4o-mini: conversational reply using history] └─ out-of-scope→ [block with explanation]

LangGraph orchestrates the pipeline with a MemorySaver checkpointer, maintaining per-session conversation history via a thread_id. This enables natural follow-up questions ("what do you mean by hour 0?") to receive contextual clarifications rather than SQL queries. The backend is FastAPI; the UI is Next.js branded as Sele the Analyst.

2. Key Engineering Decisions

DecisionRationale
Chunked CSV loading Prevents OOM on large files. Same code handles 4M or 400M rows without modification.
PostgreSQL COPY over INSERT 80× faster bulk insert — 2 minutes vs ~70 minutes for 4.2M rows.
Pre-aggregation table Daily/type analytics scan 45 rows instead of 4.2M — zero query-time aggregation cost.
Text-to-SQL vs vector RAG Structured data lives in a relational database. SQL gives exact answers; vector similarity gives approximate document matches.
Hallucination detection Back-translation re-reads the generated SQL and scores alignment with the original question 0–1. Final confidence = (LLM self-confidence + alignment) / 2.
Read-only DB transaction BEGIN READ ONLY … ROLLBACK ensures no SQL generated by the LLM can ever modify data, even if guardrails fail.
LangGraph orchestration Separates routing, SQL generation, execution, validation, and response into discrete, testable nodes. MemorySaver provides per-session conversation memory.
Docker multi-service stack Reproducible, dependency-free setup. Four services start in correct order via health checks. Evaluators run one command.

3. Challenges & How I Solved Them

ChallengeSolution
Out-of-memory on 4.2M rows Chunked reading with chunksize=100_000 — memory stays flat throughout.
Fraud SUM returning NULL Transaction types with zero fraud rows return NULL from SUM(amount) FILTER (WHERE is_fraud). Fixed with COALESCE(…, 0).
LLM querying wrong table The model used transaction_summary.transaction_hour (column does not exist). Fixed by adding explicit TABLE SELECTION RULES and a targeted few-shot example to the schema context.
Gibberish returning results Three-layer guard: UI pre-validation, LLM NOT_ANSWERABLE signal, Python regex blocking unsafe SQL keywords.
Follow-up questions blocked Added LangGraph classify node: detects follow-up vs data query and routes to a clarify node that responds using conversation history.
Docker healthcheck failure python:3.11-slim has no curl. Replaced with python -c "urllib.request.urlopen(...)".

4. Assumptions

  • Step = simulation hour: The dataset step column represents elapsed hours. step % 24 gives hour of day; (step // 24) + 1 gives simulation day (1–9).
  • Single currency: No currency conversion applied — all amounts are treated as one local unit, consistent with M-Pesa simulation data.
  • Fraud label is ground truth: The isFraud column is taken as authoritative. No additional fraud detection model was built.
  • Static dataset: The pipeline is idempotent (TRUNCATE … RESTART IDENTITY) but does not handle real-time ingestion — outside the scope of this assessment.
  • OpenAI API available: The RAG application requires an OpenAI API key and internet access. The assessment environment was assumed to have both.
  • Demo-scale memory: MemorySaver stores conversation state in-memory per server process. Production would use a Redis or PostgreSQL-backed checkpointer for multi-user, persistent history.
